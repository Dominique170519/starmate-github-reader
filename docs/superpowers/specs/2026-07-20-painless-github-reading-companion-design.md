# 星伴读插件：无痛 GitHub 阅读伴侣设计

日期：2026-07-20

## 1. 产品目标

面向没有技术背景、容易被专业名词劝退的读者，把 GitHub 仓库和项目文档变成可以持续阅读、理解、回顾和追踪更新的学习空间。

成功标准不是“生成一份摘要”，而是让用户：

- 不离开原文即可理解当前术语。
- 随时知道自己读到哪里、还要读多久。
- 下次回来能接着读，并看见作者更新了什么。
- 阅读一篇文章时就能形成知识图谱；阅读多篇后自动形成跨文档关系。
- 每条解释和图谱关系都能回到原文证据。

首版继续不接入模型 API。术语解释、关系识别和更新摘要由规则、词典、原文结构及 GitHub 元数据生成；后续可以增加可选模型能力，但不能让基础阅读依赖模型。

## 2. 范围与非目标

### 本轮范围

- GitHub 仓库 README、Markdown 文件。
- GitHub Pages 中的 Docsify 文档站，首先覆盖 Datawhale `hello-agents`。
- 专业名词轻提示与三段式解释。
- 阅读进度、预计剩余时间、阅读明细和回看标记。
- 作者更新明细与内容差异。
- 单篇及跨文档知识图谱动态生成。
- 插件侧栏小图谱与 Web App 完整图谱联动。

### 暂不包含

- 自动运行任意仓库代码。
- 对所有普通网页提供通用阅读支持。
- 没有原文证据的自由生成式解释。
- 首版跨浏览器、跨设备账号同步。Chrome 本地数据先可完整使用；Web App 在可用云存储环境中继续保留云端知识库。

## 3. 核心体验

### 3.1 顶部阅读进度条

插件启动后，在页面顶部显示一条不遮挡原站导航的细进度条：

- 当前章节阅读百分比。
- 当前章节名称。
- 根据剩余正文长度估算的剩余时间。

进度只计算正文阅读区域。切换 Docsify 章节后重置到新章节并恢复该章节历史位置。用户主动滚动、选择文本或停留在可见页面时才累计有效阅读时间；后台标签页不计时。

### 3.2 专业名词轻提示

每个章节最多提示六个高价值术语，只标记首次出现位置，使用低对比度虚线，避免满屏高亮。

点击或触摸术语后显示一张小卡片：

1. **简单解释**：一句不依赖其他专业名词的话。
2. **生活类比**：用熟悉场景建立直觉。
3. **在这段话里的作用**：结合当前段落说明作者为什么在这里使用它。

卡片提供“我懂了”和“需要回看”。后者进入阅读明细与复习列表。解释必须记录原文段落位置和解释来源；规则词典无法可靠解释时，显示“暂未收录”，不编造内容。

### 3.3 侧栏阅读明细

侧栏分为四个入口：

- **本章**：目录、当前章节目标、进度和剩余时间。
- **术语**：本章术语、已理解和需要回看的状态。
- **图谱**：当前文档的项目—章节—概念小型图谱。
- **更新**：上次阅读后新增、修改和删除的内容。

阅读明细记录：累计有效阅读时间、最后章节、章节完成度、查看过的术语、回看项及最近阅读时间。

## 4. 页面适配器

插件不再把所有页面当成同一种 GitHub 页面，而是通过独立适配器统一输出正文信息。

每个适配器输出：项目标识、文档标识、原文地址、仓库地址、当前版本、章节、正文段落、代码块和当前路由。

### GitHub 仓库适配器

- 识别 `github.com/{owner}/{repo}` 及仓库内 Markdown 文件。
- 读取 `.markdown-body`、README 和文件路径。
- 使用仓库默认分支与当前文件版本作为更新依据。

### Docsify / GitHub Pages 适配器

- 用户首次在某个 GitHub Pages 站点启用插件时，申请该站点访问权限。
- 识别 `article.markdown-section` 正文。
- 监听 `hashchange` 和正文容器更新，切换章节后重新生成目录、术语和进度。
- 将 Docsify 哈希路由映射到仓库中的 Markdown 路径。
- Datawhale 页面映射到 `datawhalechina/hello-agents` 仓库。

如果页面无法映射到公开仓库，阅读进度和本页术语仍可工作，但作者更新与提交明细显示为不可用。

## 5. 作者更新与动态刷新

保存文档时记录一个 `DocumentSnapshot`：版本标识、文件路径、章节标题、章节内容指纹、更新时间和来源提交。

检查更新有三个入口：

- 打开已保存文档时检查一次。
- Chrome 后台每天检查一次收藏文档。
- 用户手动点击“立即检查”。

更新引擎比较新旧快照，输出：

- 新增章节。
- 修改章节。
- 删除或移动章节。
- 最新提交时间、作者、提交说明和 GitHub 链接。

GitHub API 限流时保留旧快照，显示下次重试时间，不清空现有记录。相同版本不重复生成更新事件。

## 6. 知识图谱

### 6.1 单篇文档立即生成

图谱不再等待两篇文档。第一篇文档加入后立即生成：

`项目 → 包含 → 文档 → 包含 → 章节 → 解释 → 概念`

章节和概念节点均携带原文位置。点击节点回到对应章节；点击关系显示形成关系的原文依据。

### 6.2 多篇文档形成关系

第二篇文档加入后，概念先经过大小写、单复数和已知别名归一化，再建立跨文档关系：

- 共同概念。
- 前置知识。
- 相似实现。
- 不同观点。
- 实践或证据。

只有找到原文依据时才建立关系。无法解释的关系不显示。

### 6.3 更新推动图谱变化

内容更新后按差异增量修改图谱：

- 新增内容创建节点和关系。
- 修改内容更新节点并标记“上次阅读后变化”。
- 删除内容进入归档，不立即抹除用户笔记和阅读记录。
- 受影响的跨文档关系重新计算，未受影响部分保持不变。

插件侧栏只展示当前文档的局部图谱；Web App 展示完整知识图谱、跨文档关系和更新历史。

## 7. 数据边界

核心数据分为五类，彼此独立：

- `DocumentSnapshot`：原文结构和版本。
- `TermExplanation`：术语、简单解释、类比、上下文作用和证据。
- `ReadingSession`：有效阅读时间、进度和最后位置。
- `UpdateEvent`：新增、修改、删除和提交信息。
- `GraphNode` / `GraphEdge`：图谱节点、关系和证据。

Chrome 插件使用 `chrome.storage.local` 保存设备数据，并限制快照与历史数量。Web App 保留现有设备存储与可用云端存储模式。任何存储失败都不能阻塞当前页面阅读。

## 8. 错误与降级

- 页面不受支持：插件明确显示原因，不展示空白侧栏。
- Docsify 内容尚未加载：等待正文容器出现后初始化，不重复创建侧栏。
- 仓库映射失败：保留本页进度和术语，隐藏作者更新。
- 术语解释不确定：显示未收录，不生成猜测。
- GitHub API 限流：使用缓存并延迟重试。
- 图谱关系没有证据：不连线，并解释“暂未找到可靠关系”。
- 页面结构改变：单个适配器失败不能影响其他站点。

## 9. 验收标准

### GitHub 仓库

- 打开 README 后出现进度条和侧栏。
- 滚动时章节进度、剩余时间和最后位置正确更新。
- 一个术语只在本章首次出现处轻提示，卡片包含三段解释。
- 单篇 README 可以生成项目—章节—概念图谱。

### Datawhale Docsify 页面

- 用户授权站点后插件可以在目标页面启动。
- 切换哈希章节后，目录、进度、术语和局部图谱同步更新。
- 当前章节能映射到仓库 Markdown，并显示提交与更新信息。

### 动态更新

- 相同版本重复检查不会产生重复事件。
- 增加、修改、删除章节时生成正确差异。
- 图谱只更新受影响节点和关系。
- GitHub API 失败时旧数据和阅读记录仍可使用。

### 隐私与可用性

- 插件只在用户启用的技术文档站点运行。
- 不上传用户选择的原文、笔记或阅读记录到未声明服务。
- 键盘与触摸设备均可打开术语解释并操作侧栏。

## 10. 实施边界建议

为了降低一次性改动风险，实施顺序为：

1. 拆出页面适配器，并修复 GitHub Pages / Docsify 支持。
2. 增加术语解释、进度条和阅读明细。
3. 建立单篇图谱，再扩展跨文档关系。
4. 增加快照、作者更新、后台检查和图谱增量更新。
5. 将插件局部图谱与 Web App 完整图谱联动。

每一步都必须保留当前目录、原文搜索、笔记和导入 Web App 的已有能力。
