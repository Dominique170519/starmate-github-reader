# 有证据的实践案例板块设计

## 背景与目标

技术小白连续阅读理论内容时，容易因为看不到真实用途而疲劳。实践案例板块要在不牺牲可信度的前提下，让读者知道“这个项目能做出什么”，并把案例转成可以在星伴读内开始的小任务。

第一版只展示有明确证据的案例。宁可显示“暂无可信案例”，也不把概念相似的项目称为当前项目的应用案例。

## 范围

第一版支持两类案例：

1. **官方实践**：当前仓库内路径或标题明确属于 `example`、`examples`、`demo`、`showcase`、`sample`、`samples`、`cookbook`、`tutorial` 或 `quickstart` 的内容。
2. **明确使用**：其他公开 GitHub 仓库的 README 明确包含当前仓库的完整 `owner/name` 或完整 GitHub URL，并提供可打开的证据链接。

不包含仅凭关键词、语言、主题或相似描述推断的“相关项目”。不使用模型补写不存在的案例事实。

## 用户体验

全局地图增加第 5 个模式“实践案例”。内容按当前收藏仓库分组，不把不同项目的案例混为一谈。

每张案例卡片展示：

- 案例名称和证据等级；
- 它做了什么，以及为什么值得技术小白查看；
- 建议先看的文件或章节；
- 证据来源和原始链接；
- 预计体验时间；
- “开始复刻”入口。

“开始复刻”在 App 内创建三步轻任务：先观察可见结果，再回到对应原理，最后记录一次自己的改动或理解。任务状态保存在本地学习记录中；开启云同步后可进入用户学习数据。

如果没有可信案例，页面显示明确空状态，并提供仓库内官方目录入口和 GitHub 精确搜索入口。搜索失败或达到 GitHub 频率限制时，保留已验证的官方实践，外部案例区域显示失败原因，不用缓存内容冒充最新结果。

## 架构与数据流

案例识别分为三个边界清晰的单元：

1. `practice-case` 纯函数模块从学习包的文件、章节和链接中识别官方实践并生成证据对象。
2. `/api/practice-cases` 在用户打开案例板块时查询 GitHub，验证外部 README 是否明确引用当前仓库，并返回标准案例对象。
3. 页面组件按仓库渲染结果、空状态、错误状态和复刻任务，不自行判断证据可信度。

标准案例对象至少包含 `id`、`sourceRepository`、`title`、`kind`、`summary`、`recommendedEntry`、`evidence`、`estimatedMinutes` 和 `verifiedAt`。`kind` 只允许 `official` 或 `explicit-reference`。

学习包按作者更新重新生成后，官方案例会重新计算。外部案例按需刷新，并记录验证时间。案例本身不自动成为知识图谱事实；只有案例证据中出现可定位的章节或概念时，才可建立带证据的关系。

## 安全与限制

- 只读取公开 GitHub 内容。
- GitHub Token 只在服务端使用，不返回浏览器。
- 查询按 IP 和仓库限流，并对仓库名称、URL 和响应结构做校验。
- 外部引用被删除后，下次验证应移除该案例，而不是永久保留。

## 测试与验收

- 能识别常见官方案例目录，且不会把普通文档误判为案例。
- 外部案例必须包含完整仓库标识或 URL 才能通过。
- 每个案例都有可打开的证据链接和验证时间。
- GitHub 限流、网络失败和零结果都有独立可理解的界面。
- “开始复刻”能在 App 内生成并保存三步任务。
- 新增或删除官方案例文件后，刷新学习包会改变案例结果。
